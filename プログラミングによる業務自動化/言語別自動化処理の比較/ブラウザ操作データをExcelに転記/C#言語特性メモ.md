# C# 言語特性メモ（ブラウザ操作 → Excel 転記）

VBA との比較を前提として、C# の強み・弱み・書きやすい部分・書きにくい部分を整理する。

---

## C# の強み

- **型安全**: セレクタ取得結果・セル値を型付きオブジェクトで扱えるため、実行時の型エラーが減る。VBA は Variant 多用で実行時エラーが起きやすい。
- **IDE 補完**: Visual Studio / Rider のインテリセンスが強力。Selenium の API も補完対象になる。
- **async/await**: ページ読み込み待機を非同期で書ける。複数ページを並行取得する場合（`Task.WhenAll`）にも自然に拡張できる。
- **LINQ**: テーブル行をオブジェクトに射影・フィルタ・変換を宣言的に書ける。ループのネストが深くなりにくい。
- **NuGet エコシステム**: EPPlus・ClosedXML など高品質な Excel 操作ライブラリが選べる。Selenium の最新 WebDriver も NuGet で取得可能。
- **単体テスト**: 処理を関数に分割しやすく、xUnit / NUnit でモック化した自動テストを書ける。

---

## C# の弱み

- **セットアップコスト**: .NET SDK インストール・プロジェクト作成・NuGet 復元が必要。VBA はファイルを開けばすぐ書ける。
- **ビルドが必要**: 変更するたびにビルドが必要（ホットリロードは限定的）。VBA は即時実行できる。
- **COM 参照のもろさ**: `Microsoft.Office.Interop.Excel` はインストール済みの Excel バージョンに依存する。PC 環境が変わると参照が壊れる。GC に任せた COM 解放は安全でないため `Marshal.ReleaseComObject` の手動管理が必須。
- **配布の難しさ**: 実行ファイルを渡す場合は .NET ランタイムの有無を意識する必要がある。VBA ならマクロ入りファイルを渡すだけ。
- **ChromeDriver バージョン管理**: Chrome 本体と ChromeDriver のバージョンを合わせる必要がある。`Selenium.Manager` (Selenium 4.6+) で自動取得できるが、社内ネットワーク制限がある場合は手動配置が必要。

---

## 効率的に書ける部分

- **LINQ によるデータ変換**: `FindElements` の結果をそのまま `.Select().ToList()` で DTO に変換できる。VBA のような For Each + インデックス管理が不要。
- **async/await による並列取得**: ページネーションがある場合、全ページ URL を先に収集して `Task.WhenAll` で並行フェッチすると直列より大幅に速くなる。
- **EPPlus の一括書き込み**: `LoadFromCollection<T>()` を使うと DTO リストをそのままシートに展開できる。
- **COM の配列一括代入**: `Range.Value2 = object[,]` で 2D 配列をまとめてセットすると、セル単位の代入より数十〜数百倍速い（VBA でも同じイディオムが有効）。

---

## 非効率になりがちな部分

- **COM Interop のセル単位書き込み**: `ws.Cells[r, c].Value = ...` をループで回すと COM のラウンドトリップが行数分発生し、数百行で体感できる遅さになる。2D 配列一括代入で回避する。
- **WebDriverWait の乱用・省略両方がコスト**: `Thread.Sleep` での固定待機は遅い。かといって `wait.Until` の条件が緩いとフレームリロード後に StaleElementReferenceException が出る。要素の再取得パターンを統一しておく必要がある。
- **例外処理の肥大化**: Selenium + COM の組み合わせは例外の種類が多い。握りつぶしすぎると原因追跡が困難になる。

---

## VBA との比較視点

| 観点 | C# | VBA |
|------|----|----|
| 実行までの手間 | プロジェクト作成・ビルド必要 | ファイルを開いてすぐ実行 |
| 型安全 | コンパイル時に検出 | 実行時 Variant エラーが多い |
| 非同期処理 | async/await で自然に書ける | 標準では同期のみ |
| Excel 操作 | EPPlus（COM 不要）が使える | Excel オブジェクトモデルを直接操作 |
| ブラウザ操作 | Selenium の公式サポートが厚い | SeleniumWrapper など非公式が中心 |
| 保守性 | クラス分割・テスト容易 | モジュールが大きくなりがち |
| 配布 | ランタイム同梱 or .NET 前提 | マクロ入り Excel ファイルを渡すだけ |

VBA が有利なのは「今すぐ動かせる」「配布が簡単」「Excel オブジェクトモデルとの親和性」。
C# が有利なのは「大規模化・保守」「非同期・並列」「COM に依存しない Excel 操作」。
