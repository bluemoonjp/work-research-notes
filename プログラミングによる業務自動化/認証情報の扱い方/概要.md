# 社内で配布・実行するプログラムでの認証情報の扱い方（C#）

## はじめに

社内システムへのアクセスに必要な認証情報（ID・パスワード・アクセストークン）をプログラムに持たせる方法はいくつかある。ソースコードや設定ファイルに平文で書くと、リポジトリへの誤コミットや配布ファイルの漏洩時に認証情報がそのまま露出するリスクがある。

なお、サーバーがオンプレミスに設置されていても、Azure Arc を経由して Azure Key Vault を利用することは可能である。「オンプレサーバー＝クラウド連携なし」ではない点に注意する。

## 設計上の原則

実装方法を選ぶ前に、以下の原則を確認する。

- **共有シークレットを社員PCに配布しない**: 共有IDやアクセストークンを社員PC上のプログラムに持たせると、1台の漏洩が全体の漏洩につながる。可能であれば、サーバー側で代理実行するか、OAuth などの利用者単位のトークンを使う設計を優先する。
- **中央管理・ローテーションできる方法を優先する**: 認証情報は定期的な変更が必要。変更手順が煩雑だと放置されやすい。
- **最小権限の原則**: プログラムに与える認証情報のスコープは必要最小限にする。

## 社員PC向けの方法

### 方法1: Windows DPAPI (CurrentUser) — 個別認証情報

**仕組み**: Windows が提供する `ProtectedData.Protect()` を使い、現在のWindowsユーザーの資格情報を鍵として暗号化する。暗号化したデータはそのWindowsユーザーとそのPCでしか復号できない。

**向いている場面**: 社員が自分のID・パスワードを入力して使うプログラム。

**メリット**:
- 暗号化はOSが担当するため、鍵管理が不要
- 初回入力後は自動で復号されるので利便性が高い
- 他のWindowsユーザーからは復号できない

**注意事項**:
- Windowsのユーザーパスワードを変更すると、既存の暗号化データが復号できなくなる場合がある（その場合は再入力が必要）
- `%LOCALAPPDATA%` に保存する（`%APPDATA%` だとローミングプロファイルで他のPCにコピーされる恐れがある）
- .NET CoreはWindowsのみ対応
- 詳細な実装例: [業務システムからExcelダウンロードして集計（DPAPI版）](../言語別自動化処理の比較/業務システムからExcelダウンロードして集計/2_業務システムからExcelダウンロードして集計_DPAPI_CSHARP.cs)

### 方法2: Windows資格情報マネージャー — 共有認証情報の配布

**仕組み**: Windowsに組み込まれた資格情報ストアを利用する。`CredWrite` / `CredRead` (Win32 API) を P/Invoke で呼び出す。

**向いている場面**: ITが事前に社員PCへ認証情報を配布しておき、プログラムが読み取る用途。

**ITによる事前登録の方法**:
```
cmdkey /add:MyApp:ApiToken /user:service-account /pass:アクセストークン
```
ログオンスクリプトやグループポリシー（GPO）で各社員のPCに実行する。

**メリット**:
- プログラムが認証情報を保存・管理する必要がない
- NuGetパッケージ不要（Win32 APIを直接呼び出す）

**注意事項**:
- 資格情報マネージャーは**ユーザー単位の保存**。全ユーザーで共有するストアではない
- コントロールパネルの「資格情報マネージャー」から中身が見えるため、高権限の認証情報には不向き
- 1件あたりの保存サイズ上限は**2560バイト**。長いアクセストークンやJSONは注意が必要
- .NETのPlatform: Windows専用

## サーバー向けの方法 — Azure Key Vault

コードに認証情報を一切持たない最も堅牢な構成。Azure Key Vault にシークレットを集中管理し、アプリケーションは Managed Identity で取得する。

オンプレミスサーバーでも Azure Arc 対応サーバーとして登録すれば同じ方法が使える。完全にAzureへの接続を持たない閉域網環境では後述のオンプレ向け方法を選ぶ。

### 方法5: Managed Identity + Azure Key Vault

**仕組み**: Azure Key Vault にシークレットを保存し、プログラムは `ManagedIdentityCredential` でKey Vaultから取得する。コードに認証情報を一切持たない。

**向いている場面**:
- **Azure VM / App Service / Azure Functions**: そのままManaged Identityを使用できる
- **オンプレサーバー（Azure Arc登録済み）**: Azure Arcエージェント (`azcmagent`) を導入してサーバーをAzure Arcに登録することで、オンプレサーバーにもManaged Identityが付与される

**メリット**:
- コードに認証情報が残らない
- シークレットのローテーション・アクセスログが集中管理できる
- Azure Arc を利用すれば既存のオンプレサーバーにも適用できる

**注意事項**:
- オンプレサーバーには Azure Arc 登録（`azcmagent` の導入）が前提
- 完全にAzureへの接続を持たない閉域網環境では利用できない

NuGet: `Azure.Identity`、`Azure.Security.KeyVault.Secrets`

## サーバー向けの方法 — 完全オンプレ

Azure Key Vault が使えない環境（閉域網・Azure非契約など）向けの選択肢。

### 方法6: HashiCorp Vault — 複数台・集中管理

**仕組み**: オンプレ環境に HashiCorp Vault サーバーを構築し、各アプリケーションがVaultに認証してシークレットを取得する。

**向いている場面**: Azure非依存でオンプレ完結・複数サーバーのシークレットを集中管理したい場合。

**認証方式（オンプレAD環境での主な選択肢）**:
- **AppRole認証**: アプリケーション単位で `role_id` + `secret_id` を使ってVaultに認証する
- **LDAP認証**: Active DirectoryをバックエンドにしてVaultに認証する（ADパスワードをアプリに固定保存する設計は避け、実行基盤側から注入する）

**メリット**:
- Azure非依存でオンプレ完結
- シークレットのローテーション・アクセス制御・監査ログを集中管理できる

**注意事項**:
- Vaultサーバー自体の構築・運用（初期化・unseal・バックアップ・監視）が必要
- AppRoleの `secret_id` 配布経路の設計が必要

NuGet: `VaultSharp`

### 方法3: 環境変数 — 単一台でシンプルに

**仕組み**: `Environment.GetEnvironmentVariable()` でOSの環境変数を読み取る。認証情報はプログラムの外（OSの設定またはCI/CDのシークレット）に置く。

**向いている場面**: タスクスケジューラやCI/CDから実行するサーバーバッチ処理。

**メリット**:
- シンプルで実装が容易
- Linux・Docker でも同じコードが動く（ポータブル）
- CI/CDのシークレット機能（GitHub Actions Secrets等）と相性が良い

**注意事項**:
- 環境変数は**平文で保存**される。OSの環境変数に直接書き込むのは最終手段。理想は実行時にCI/CDやサービスマネージャーから注入する
- WindowsのMachine環境変数はシステム管理者ならレジストリから読める

**Windows Serverでの設定方法**（管理者が実行）:
```powershell
[System.Environment]::SetEnvironmentVariable("APP_API_TOKEN", "トークン値", "Machine")
# 設定後はサービス or タスクスケジューラを再起動する
```

### 方法4: DPAPI (LocalMachine) + 暗号化設定ファイル — 環境変数が使えない場合

**仕組み**: `ProtectedData.Protect()` を `DataProtectionScope.LocalMachine` で使い、認証情報を暗号化してファイルに保存する。管理者が初回セットアップ時のみ実行し、以降は自動で復号して使う。

**向いている場面**: 環境変数を設定できないサーバー環境や、複雑な設定が必要な場合。

**重要な制限（必ず確認）**:
- 同一マシン上の**任意のプロセスが復号可能**（管理者権限の別プロセス含む）
- 暗号化ファイルを別のサーバーにコピーしても復号できない（**マシン間で移植不可**）
- 複数台展開・コンテナ・サーバー再構築が多い環境には向かない

**ACLによるファイル保護**（セットアップ後に必ず実施）:
```cmd
icacls credentials.dat /inheritance:r /grant "NT AUTHORITY\SYSTEM:R" /grant "BUILTIN\Administrators:R"
```

## 補足（参考）

### 証明書認証（クライアント証明書 / mTLS）

ID・パスワードやAPIキーの代わりにクライアント証明書で認証する方式。Windows証明書ストアに格納し、秘密鍵にACLをかけることでプログラムが平文のシークレットを持たずに済む。接続先サービスがクライアント証明書認証（mTLS等）に対応している場合に選択肢となる。証明書の発行・更新・失効管理の運用が必要。

### gMSA（グループ管理サービスアカウント）— Windows統合認証専用

Windows Serverのドメイン環境でWindowsサービスやタスクスケジューラを動かす際の **サービスアカウント管理** に使う仕組み。Windowsがパスワードを自動ローテーションし、サービスの実行アカウントを人が管理する必要がなくなる。

**重要な制限**: gMSAで「不要になる認証情報」はWindowsサービス自体のドメインログオン資格情報のみ。外部REST APIのAPIキー・アクセストークン・データベースパスワードなど、接続先サービスが要求するシークレットは別途管理が必要。gMSAが直接役立つのは、SQL Server Windows認証・ファイル共有など **Windows統合認証（Kerberos/NTLM）が使える接続先に限られる**。

## 共通の注意事項

- **ログ・例外メッセージに認証情報の値を出力しない**: デバッグ時も含めて徹底する
- **バージョン管理から除外**: `credentials.dat` などの保存ファイルは `.gitignore` に追加する
- **ローテーション計画**: 認証情報を変更する手順を事前に決めておく。プログラムが認証情報の有効期限切れを検知して再入力を促す仕組みがあると望ましい
- **削除手順の周知**: 離職者やトークン失効時に認証情報を削除する手順を文書化しておく
