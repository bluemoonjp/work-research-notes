# 非同期処理（async / await）

## この記事で学ぶこと

- 同期・非同期・並列の違い
- `async` / `await` の基本的な使い方
- `Task` / `Task<T>` / `ValueTask` の使い分け
- `async void` を避けるべき理由
- `ConfigureAwait(false)` の意味
- `CancellationToken` によるキャンセル
- `Task.WhenAll` / `Task.WhenAny` で複数タスクを制御する
- よくある落とし穴（デッドロック・`await` の忘れなど）

## 同期・非同期・並列の違い

| | 説明 | 向いている処理 |
|---|---|---|
| **同期（Synchronous）** | 処理が終わるまで呼び出し元をブロック | シンプルな計算処理 |
| **非同期（Asynchronous）** | 処理を待つ間、呼び出し元に制御を返す | I/O 待ち（ファイル・ネットワーク・DB） |
| **並列（Parallel）** | 複数スレッドで同時に処理する | CPU バウンドな重い計算 |

`async` / `await` は**非同期 I/O**に適しています。スレッドを占有せず待機するため、スケーラビリティが高まります。CPU バウンドな処理の並列化には `Parallel.For` や `Task.Run` を使います。

## async / await の基本

```csharp
// async を付けると await を使えるメソッドになる
public async Task<string> FetchDataAsync(string url)
{
    using var client = new HttpClient();

    // await: 非同期操作が完了するまで待ちつつ、スレッドは解放する
    string content = await client.GetStringAsync(url);

    return content;
}

// 呼び出し側も await する
string data = await FetchDataAsync("https://example.com");
```

**`await` の仕組み:**  
`await` に達すると、メソッドは一時停止し呼び出し元に制御が戻ります。非同期操作が完了すると、`await` の次の行から処理が再開されます。

## Task / Task<T> / ValueTask

| 型 | 使いどころ |
|---|---|
| `Task` | 値を返さない非同期処理（`void` の代わり） |
| `Task<T>` | 値を返す非同期処理 |
| `ValueTask<T>` | ほぼ同期的に完了する（キャッシュ済みなど）高頻度呼び出しの最適化 |

```csharp
public async Task SaveAsync(string data) { /* ... */ }      // 戻り値なし
public async Task<int> CountAsync()      { /* ... */ return 42; } // int を返す

// ValueTask: 結果がすでにある場合はヒープ確保なしで返せる
public ValueTask<int> GetCachedAsync()
{
    if (_cache.TryGetValue("key", out int val))
        return ValueTask.FromResult(val); // Task を生成しない
    return new ValueTask<int>(FetchFromDbAsync());
}
```

`ValueTask` は最適化用途で、通常は `Task<T>` を使えば十分です。

## async void を避けるべき理由

```csharp
// NG: async void はイベントハンドラ以外では使わない
async void LoadDataWrong()
{
    await Task.Delay(1000);
    throw new Exception("このエラーは捕捉できない"); // アプリがクラッシュする
}

// OK: async Task にする
async Task LoadDataCorrect()
{
    await Task.Delay(1000);
    throw new Exception("このエラーは呼び出し元で catch できる");
}
```

`async void` にすると:
1. 呼び出し元が `await` できないため、例外を捕捉できない
2. 完了を待てないため、テストが難しい

**唯一の例外:** WinForms・WPF のイベントハンドラ（シグネチャが `void` に固定されている場合）。

```csharp
// WinForms のイベントハンドラは void が必要（やむを得ない）
private async void Button_Click(object sender, EventArgs e)
{
    await DoWorkAsync();
}
```

## ConfigureAwait(false)

```csharp
// ライブラリコードでは ConfigureAwait(false) を使う
public async Task<string> LibraryMethodAsync()
{
    // false: 再開時に元の同期コンテキストに戻らない
    var result = await HttpClient.GetStringAsync(url).ConfigureAwait(false);
    return result;
}
```

| | `await task` | `await task.ConfigureAwait(false)` |
|---|---|---|
| 再開コンテキスト | 元のコンテキスト（UI スレッドなど） | スレッドプールの任意スレッド |
| デッドロックリスク | あり（UI スレッドで `.Result` と組み合わせると） | 低い |
| 用途 | アプリコード（UI 操作が必要な場合） | ライブラリコード |

UI コードでは `await` 後に UI を操作することがあるため、通常は `ConfigureAwait(false)` を付けません。ライブラリコードでは不要なコンテキスト切り替えを避けるために付けます。

## CancellationToken によるキャンセル

```csharp
public async Task ProcessAsync(CancellationToken cancellationToken = default)
{
    for (int i = 0; i < 100; i++)
    {
        // キャンセルが要求されたら OperationCanceledException をスロー
        cancellationToken.ThrowIfCancellationRequested();

        await DoStepAsync(i, cancellationToken); // token を伝播させる
    }
}

// 呼び出し側
using var cts = new CancellationTokenSource();
cts.CancelAfter(TimeSpan.FromSeconds(5)); // 5 秒後に自動キャンセル

try
{
    await ProcessAsync(cts.Token);
}
catch (OperationCanceledException)
{
    Console.WriteLine("処理がキャンセルされました。");
}
```

**`CancellationToken` は引数として伝播させる**のが基本原則です。使わないメソッドでも受け取って下位に渡すことで、全体のキャンセルが機能します。

## Task.WhenAll / Task.WhenAny

```csharp
// Task.WhenAll: すべてのタスクが完了するまで待つ（並列実行）
var task1 = FetchAsync("url1");
var task2 = FetchAsync("url2");
var task3 = FetchAsync("url3");

string[] results = await Task.WhenAll(task1, task2, task3);
// 3 つが同時に実行され、すべて完了したら結果が揃う

// Task.WhenAny: 最初に完了したタスクを返す（タイムアウトパターンなど）
var fetchTask   = FetchAsync("url");
var timeoutTask = Task.Delay(TimeSpan.FromSeconds(5));

Task completed = await Task.WhenAny(fetchTask, timeoutTask);
if (completed == timeoutTask)
{
    Console.WriteLine("タイムアウトしました。");
}
else
{
    string data = await fetchTask; // すでに完了しているので即座に返る
}
```

`Task.WhenAll` で 1 つでもタスクが例外をスローすると `AggregateException` がスローされます（`await` すると最初の例外がアンラップされます）。

## よくある落とし穴

### 落とし穴 1: デッドロック（`.Result` / `.Wait()` の使用）

```csharp
// NG: UI スレッドや ASP.NET のコンテキストで使うとデッドロック
string data = GetDataAsync().Result;      // ブロッキング呼び出し
string data = GetDataAsync().GetAwaiter().GetResult(); // 同様

// OK: 非同期チェーンを維持する（呼び出し元も async にする）
string data = await GetDataAsync();
```

### 落とし穴 2: await を忘れる

```csharp
// NG: await を忘れると Task が返るだけで処理を待たない
public async Task ProcessAsync()
{
    SaveAsync(); // ← await を忘れた（コンパイラ警告が出る）
    Console.WriteLine("完了"); // SaveAsync が終わる前に実行される
}

// OK
public async Task ProcessAsync()
{
    await SaveAsync();
    Console.WriteLine("完了");
}
```

### 落とし穴 3: ループ内で Task を待ち忘れる

```csharp
// NG: 全タスクを同時に開始せず、順番に 1 つずつ待つ
foreach (var url in urls)
{
    await FetchAsync(url); // 直列実行（並列にならない）
}

// OK: 並列に実行して WhenAll で待つ
var tasks = urls.Select(url => FetchAsync(url));
await Task.WhenAll(tasks);
```
