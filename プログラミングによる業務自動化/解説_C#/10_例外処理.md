# 例外処理

## この記事で学ぶこと

- `try` / `catch` / `finally` / `when` の構文と使い方
- 例外クラスの階層構造
- カスタム例外クラスの作り方
- `throw` と `throw ex` の違い（スタックトレースの保持）
- 例外フィルター（`catch (Exception e) when (...)`）
- よく遭遇する例外型とその原因
- 例外を使うべき場面と避けるべき場面

## try / catch / finally

```csharp
try
{
    // 例外が発生する可能性のある処理
    int result = Divide(10, 0);
}
catch (DivideByZeroException ex)
{
    // 特定の例外を捕捉
    Console.WriteLine($"ゼロ除算: {ex.Message}");
}
catch (Exception ex)
{
    // それ以外の例外（より上位）
    Console.WriteLine($"予期しないエラー: {ex.Message}");
}
finally
{
    // 例外の有無にかかわらず必ず実行される
    // ファイル・接続のクローズなどリソース解放に使う
    Console.WriteLine("finally 実行");
}
```

**finally の役割:**  
例外が発生してもしなくても必ず実行されます。ただし、リソースの解放には `using` 文の使用が推奨されます（`finally` で `Dispose()` を手動で呼ぶより簡潔）。

```csharp
// using: IDisposable なオブジェクトを自動的に Dispose する
using var stream = new FileStream("file.txt", FileMode.Open);
// ブロックを抜けると自動で stream.Dispose() が呼ばれる
```

## 例外クラスの階層

```
Exception
├── SystemException
│   ├── NullReferenceException    ← null を参照した
│   ├── IndexOutOfRangeException  ← 配列の範囲外アクセス
│   ├── InvalidCastException      ← 不正なキャスト
│   ├── InvalidOperationException ← オブジェクトの状態が操作に適していない
│   ├── ArgumentException         ← 引数が不正
│   │   ├── ArgumentNullException
│   │   └── ArgumentOutOfRangeException
│   ├── ArithmeticException
│   │   └── DivideByZeroException
│   ├── IOException
│   │   ├── FileNotFoundException
│   │   └── DirectoryNotFoundException
│   └── OverflowException
└── ApplicationException          ← アプリケーション固有の例外（非推奨）
```

`catch` ブロックは上から順に評価されます。**より具体的な例外を先に書く**必要があります。

```csharp
// OK: 具体的 → 抽象的の順
catch (FileNotFoundException ex) { ... }
catch (IOException ex) { ... }
catch (Exception ex) { ... }

// NG: Exception を先に書くと FileNotFoundException がヒットしない（コンパイルエラー）
catch (Exception ex) { ... }
catch (FileNotFoundException ex) { ... } // エラー: 到達不能なコード
```

## よく遭遇する例外型

| 例外 | 典型的な原因 |
|---|---|
| `NullReferenceException` | null のオブジェクトのメンバーを呼んだ |
| `ArgumentNullException` | null を渡してはいけない引数に null を渡した |
| `ArgumentOutOfRangeException` | 範囲外のインデックスや値を渡した |
| `InvalidOperationException` | オブジェクトが操作を受け付けない状態にある |
| `NotImplementedException` | 未実装のメソッドを呼んだ |
| `NotSupportedException` | その型ではサポートされていない操作を呼んだ |
| `FileNotFoundException` | ファイルが存在しない |
| `UnauthorizedAccessException` | アクセス権がない |
| `TimeoutException` | タイムアウトした |
| `OperationCanceledException` | `CancellationToken` でキャンセルされた |

## カスタム例外クラス

`Exception` を継承して独自の例外クラスを定義できます。  
3 つのコンストラクタを実装するのが慣習です。

```csharp
public class OrderNotFoundException : Exception
{
    public int OrderId { get; }

    public OrderNotFoundException(int orderId)
        : base($"注文 ID {orderId} が見つかりません。")
    {
        OrderId = orderId;
    }

    public OrderNotFoundException(int orderId, Exception innerException)
        : base($"注文 ID {orderId} が見つかりません。", innerException)
    {
        OrderId = orderId;
    }
}
```

カスタム例外は以下のような場合に使います。
- 呼び出し元が例外の種類に応じて処理を分けたいとき
- 例外に追加情報（`OrderId` など）を持たせたいとき

## throw vs throw ex（スタックトレースの保持）

**`throw`**: 元のスタックトレースを保持して例外を再スロー  
**`throw ex`**: スタックトレースをその時点でリセットしてしまう

```csharp
try
{
    DoSomething();
}
catch (Exception ex)
{
    Log(ex);             // ログは取る

    throw;    // OK: 元のスタックトレースを維持して再スロー
    // throw ex; // NG: スタックトレースがこの行からになってしまう
}
```

元の例外を別の例外に包みたい場合は `innerException` を使います。

```csharp
catch (SqlException ex)
{
    throw new DataAccessException("DB アクセスに失敗しました。", ex); // ex を inner に
}
```

## 例外フィルター（when 句）

`when` を使うと、例外の種類に加えて条件式でも捕捉を絞り込めます。

```csharp
catch (HttpRequestException ex) when (ex.StatusCode == HttpStatusCode.NotFound)
{
    Console.WriteLine("404 Not Found");
}
catch (HttpRequestException ex) when (ex.StatusCode == HttpStatusCode.ServiceUnavailable)
{
    Console.WriteLine("503 Service Unavailable");
}
catch (HttpRequestException ex)
{
    Console.WriteLine($"その他の HTTP エラー: {ex.StatusCode}");
}
```

`when` の条件が `false` の場合はそのブロックをスキップし、次の `catch` へ進みます。  
条件が `false` のときはスタックトレースがリセットされないという利点もあります。

## 例外を使うべき場面・避けるべき場面

### 使うべき場面

- 呼び出し元がリカバリすべき異常事態を通知する
- コントラクト（引数の前提条件）が破られた
- 外部リソース（ファイル・ネットワーク）への操作が失敗した

### 避けるべき場面（戻り値や `TryXxx` パターンで代替）

```csharp
// NG: 正常フローの分岐に例外を使う（パフォーマンス・可読性ともに悪い）
try
{
    var value = int.Parse(input);
}
catch (FormatException)
{
    // 変換できなかった場合の処理
}

// OK: TryParse で条件分岐
if (int.TryParse(input, out int value))
{
    // 成功
}
else
{
    // 失敗
}
```

例外の生成はコストが高いため、頻繁に発生することが想定されるパスには使わないようにします。
