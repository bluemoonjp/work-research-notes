# ブラウザ自動化の実装と共通化

## はじめに

「5_社内システムへの接続方法の選び方.md」で最終手段と位置づけたブラウザ自動化。API やプロトコルアクセスが使えず、やむを得ずブラウザ操作が必要な場合の実装方針と、複数のツールにまたがるコードの共通化を整理する。

## 1. C#（Playwright）vs RPA — Web 操作の比較

社内 Web システムの操作については、RPA だけでなく C# + Playwright も有力な選択肢になる。

| 観点 | C# + Playwright | RPA |
|---|---|---|
| ヘッドレス実行 | ○（他の作業を妨げない・サーバー実行可） | △（ツール依存） |
| 並列処理 | ○（複数ページを同時実行） | △ |
| バージョン管理 | ○（コードとして Git 管理できる） | △（独自形式が多い） |
| CI/CD・テスト | ○（通常の C# テストと同じ手法） | △ |
| UI 変更への耐性 | △（セレクタ設計次第） | △（同様） |
| デスクトップアプリ操作 | △（UI Automation API 経由） | ○ |
| 初期学習コスト | 中（C# + ブラウザ操作の知識が必要） | 低（GUI でフロー設計） |

**結論**: 社内 Web システムは C# + Playwright が有力。デスクトップアプリ（Win32・レガシー基幹系）は RPA が有力。

NuGet: `Microsoft.Playwright`

## 2. UI 変更への対処設計

ブラウザ自動化の最大の課題は、対象システムの画面が変わるたびにコードが壊れることである。これを最小限に抑える設計が必要。

### Page Object Model（POM）パターン

セレクタと画面操作をページ単位のクラスに集約するパターン。

```
// Before POM（セレクタがコードに散在する）
await page.ClickAsync("#btn-login");
await page.FillAsync("input[name='userId']", userId);

// After POM（操作はメソッド呼び出しに抽象化される）
await loginPage.ログイン(userId, password);
```

UI が変わったとき、修正箇所が各ページクラスの1か所に限定される。

### セレクタの書き方の指針

壊れにくいセレクタを選ぶ優先順位:

1. **`data-testid` / `aria-label` 等のテスト用属性**: 画面デザイン変更の影響を受けにくい
2. **ID / name 属性**: 一意性が高く安定しやすい
3. **`role` + テキスト**: `page.GetByRole(AriaRole.Button, new() { Name = "発注" })`
4. **CSS クラス**: デザイン変更で変わりやすいため避けた方が無難
5. **XPath**: 最終手段。階層構造に依存するため壊れやすい

## 3. コード共通化の方針

### 管理体制の現実

Bot の管理は部署単位であり、1 つのツールが複数のシステムをまたぐことも多い。「操作対象システムごとにソリューションを分ける」という整理は現実的ではない。

- 依頼元の部署が管理主体である体制は C# 移行後も変わらない
- 共通化の軸は「操作対象システム」ではなく「どのコードが本当に重複しているか」で判断する

### 重複の許容

**単純・独立したツールは重複して構わない。**

所有者が決まらない共通ライブラリは、UI 変更時に誰も修正せず全ツールが壊れたまま放置されるリスクがある。無理に共通化するより、独立したコードの方が管理しやすいことが多い。

共通化を検討するのは次の条件がそろったときに絞る:
- 同じ処理が **3 本以上** のツールに重複している
- 対象システムの **UI 変更が定期的に発生する**（変更のたびに全ツールを直す手間が大きい）
- **保守担当者を決められる**

### 高頻度システム向けクライアントライブラリ

使用頻度が特に高いシステム（例: 社内発注システム）は、専用のクライアントライブラリを作る価値がある。

```
発注システムClient/          ← クラスライブラリ (.csproj)
├── 発注システムClient.cs    ← エントリポイント（IPlaywright の初期化等）
├── Pages/
│   ├── ログインページ.cs    ← セレクタ・操作・画面遷移確認を集約
│   ├── 発注入力ページ.cs
│   └── 確認ページ.cs
└── Exceptions/
    └── 画面遷移例外.cs      ← 想定外のページが表示された場合
```

各ツールは `client.発注入力ページ.商品コード入力(code)` のように呼ぶだけで、セレクタや遷移確認のロジックを持たなくてよい。UI が変わったときは `Pages/` 内を修正するだけで全ツールに反映される。

### 共通ライブラリの配布方法

| 利用範囲 | 推奨方法 |
|---|---|
| 同一リポジトリ内の複数ツール | ソリューション内のプロジェクト参照（最もシンプル） |
| 複数リポジトリ・部署をまたぐ | GitHub Releases に DLL を添付、または プライベート NuGet（GitHub Packages 等） |

### 所有者を決める（最重要）

共通ライブラリには必ず **保守担当者を明示する**。

「誰が対象システムの UI 変更時に修正するか」が決まっていないライブラリは運用できない。担当者が決まらないなら、共通化せずに各ツールが独立して持つ方が現実的。
