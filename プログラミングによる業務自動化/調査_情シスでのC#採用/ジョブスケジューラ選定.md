# ジョブスケジューラ選定

情シス部門でジョブスケジューラを選定する際の考え方と選択肢をまとめます。

## この記事で学ぶこと

- 情シス部門の視点で現実的に求められる要件の整理
- Windows タスクスケジューラ・SQL Server Agent・JP1・OSS の比較
- 各選択肢の費用構造と向き・不向き
- スケジューラ選定を後回しにする戦略とそのメリット

## 1. ジョブスケジューラに求める要件の整理

情シス部門の視点で現実的に求められる要件は以下の通りです。

- **高可用性（HA）**: 実行サーバーが 1 台死んでもスケジュール実行が止まらない
- **失敗検知・通知**: ジョブが失敗したときにアラートが来る
- **リトライ・再実行**: 失敗後の再実行を確認・操作できる
- **管理・可視性**: 実行履歴・ログを一覧で確認できる
- **低メンテナンス**: 基盤の維持に工数をかけたくない
- **オンプレミス実行**: 社内 LAN 内での実行が必要

これらすべてを満たすことを目指すと高コストになります。自部門の規模・予算・リスク許容度に応じてどの要件を優先するかを先に決めることが選定の出発点です。

## 2. 選択肢の比較

| | Windows タスクスケジューラ | SQL Server Agent | JP1 | OSS（Rundeck 等） |
|---|---|---|---|---|
| HA | ✗（単一サーバー依存） | ◎（Always On / クラスタ） | ◎ | △（自前構築が必要） |
| 失敗検知・通知 | △（自前実装が必要） | ○ | ◎ | ○ |
| 管理 UI | △ | ○（SSMS） | ◎ | ○（Web UI） |
| 維持コスト | 低 | 低 | 低（2年目以降） | 高（自己管理） |
| 初期費用 | 無料 | 無料（SQL Server 付帯） | 高 | 無料 |
| オンプレ対応 | ◎ | ◎ | ◎ | ◎ |

## 3. 各選択肢の補足

### Windows タスクスケジューラ

無料・すぐ使えますが、HA がなく管理が分散しやすいという欠点があります。失敗検知を自前で実装する必要があります（例: exe の終了コードを確認してメール送信するラッパーを作るなど）。

「スケジュール実行の本数が少なく、失敗しても翌営業日に気づければよい」程度であれば現実的な選択肢です。初期コストをかけずに段階的に移行を始める際の暫定運用にも使えます。

### SQL Server Agent

SQL Server に付帯しており追加費用なしで利用できます。SQL Server の Always On 可用性グループやフェイルオーバークラスタリングと組み合わせれば HA を実現できます。

exe・PowerShell・T-SQL をスケジュール実行でき、失敗時のメール通知も設定可能です。管理は SSMS（SQL Server Management Studio）から行います。

SQL Server が既存環境にある場合の「隠れた選択肢」です。すでにライセンスを持っているにもかかわらず見落とされやすいため、まず確認する価値があります。

### JP1（日立）

国内企業での採用実績が多く、前述の要件をほぼすべて満たします。

**費用構造:** 初年度は初期ライセンス費用が重いですが、2年目以降は年間保守費（初期費用の 20% 前後）のみになります。

RPA ツールの年間コストと比較して、移行できるボット数次第では数年で回収できる可能性があります。「RPA ツールの解約コスト削減」と「JP1 導入」をセットで稟議に通すと費用対効果を説明しやすいです。

**導入検討のタイミング:** スケジュール実行が必要なジョブが一定数（目安: 数十本以上）ある場合に費用対効果が出やすくなります。

### OSS（Rundeck 等）

Web UI でスクリプト・exe を管理・スケジュール実行できます。無料で導入できますが、自己管理コストが高く、情シス部門の少人数体制では負担になりやすいです。

アップデート対応・障害対応・セキュリティパッチの適用を自部門で行う必要があるため、インフラ管理のリソースが確保できない場合は避けた方が無難です。

## 4. スケジューラ選定を後回しにする戦略

スケジューラを今すぐ決めなくてよい場合があります。以下の順序で進めることを検討してください。

1. まずデスクトップアプリ化を進める（手動起動ボットの移行）
2. 「本当にスケジュール実行が必要なもの」の本数が見えてくる
3. 残った本数と重要度で JP1 導入の費用対効果を判断する
4. それまでは RPA ツールをスケジューラとして温存し、ロジックだけ C# に逃がす

**この順序にすることのメリット:**

- スケジュール実行の実際の規模を過大評価しなくて済む
- JP1 の稟議を通す際に「削減できるボット数・コスト」という具体的な数値で説明できる
- スケジューラ選定に時間をかけている間も、開発・保守品質の改善を並行して進められる

「今の RPA 環境を全部一気に置き換える」ではなく、「移行できるものから順に C# に移し、スケジューラは後から最適なものを選ぶ」という段階的なアプローチが、現実的に組織に受け入れられやすい戦略です。
