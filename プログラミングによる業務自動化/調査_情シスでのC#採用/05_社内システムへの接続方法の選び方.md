# 社内システムへの接続方法の選び方

## はじめに

社内システムを自動化プログラムから操作する方法は複数あり、どれを選ぶかによって安定性・保守コスト・将来的な AI 連携のしやすさが大きく変わる。画面を操作して自動化できればそれでよい、という考え方は、長期的には保守コストが高くなりやすい。

## 接続方法の優先順位

以下の順で使える方法を確認し、できるだけ上位の方法を採用する。

### 優先度 1: API（REST / SOAP / OData）

最も安定しており、変更に強い。

- システムが REST API・SOAP・OData を公開している場合は **必ずこれを使う**
- 画面の構成が変わっても API の仕様が変わらなければコードの修正は不要
- レスポンスが構造化されているため、LLM へのデータ入力・出力の加工が容易
- C# では `HttpClient` + `System.Text.Json` で実装する

**確認方法**: ベンダーの開発者向けドキュメント、Swagger UI（`/swagger` または `/api-docs`）、社内の IT 担当者へ問い合わせ。

### 優先度 2: プロトコルアクセス

API はないが、標準プロトコルで直接接続できる場合。

| プロトコル | 用途 | C# での実装 |
|---|---|---|
| IMAP / SMTP | メールの読み取り・送信 | `MailKit` (NuGet) |
| SFTP / FTP | ファイルの取得・送信 | `SSH.NET` (NuGet) |
| DB 直接接続 | SQL Server・Oracle 等への直接クエリ | `System.Data` / `Dapper` |
| COM / Office | Excel・Outlook 等の Office 操作 | `Microsoft.Office.Interop.*` |

画面に依存しないため、画面操作より保守コストが低い。ベンダーへの確認なしに使える場合も多い。

### 優先度 3: ブラウザ自動化（最終手段）

API もプロトコルも使えない場合にのみ採用する。

- 社内 Web システム・SaaS → C# + Playwright（詳細は「6_ブラウザ自動化の実装と共通化.md」）
- デスクトップアプリ（Win32 等） → RPA ツール

UI が変わるたびにコードの修正が必要になるため、保守コストが最も高い。採用する場合は変更への耐性を設計段階から考慮する必要がある。

## API が使えない場合の対応

API がないシステムでも、以下のアプローチが取れる場合がある。

**ベンダーへの API 提供要求**: 社内基幹システムのベンダーに、データ取得・操作用の API または CSV エクスポート機能の追加を要求する。システム刷新のタイミングで交渉すると通りやすい。

**IT 部門から調達部門・情報システム部門への働きかけ**: 「このシステムに API がないと自動化が困難になる」という情報を契約更新・機能追加交渉のインプットとして提供する。

## AI 連携を見据えた設計判断

LLM を業務自動化に組み込む場合、プログラムがデータを取得・操作する能力の高さが直接 AI の活用幅に影響する。

- **API アクセス可能** → LLM がツール呼び出しで社内データを検索・更新できる（MCP サーバー化も可能）
- **プロトコルアクセス可能** → メール・ファイルを LLM が処理するパイプラインを構築しやすい
- **画面操作のみ** → LLM との連携が難しく、アドホックな自動化にとどまりやすい

接続方法の選択は、単なる「今動くか」の問題ではなく、将来の拡張性に関わる設計判断である。
