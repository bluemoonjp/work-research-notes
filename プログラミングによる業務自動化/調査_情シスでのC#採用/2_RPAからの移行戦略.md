# RPA からの移行戦略

Automation Anywhere などの RPA 環境から C# へ段階的に移行する戦略をまとめます。

## この記事で学ぶこと

- RPA が「唯一の実行基盤」になってしまう問題の構造
- ボットを性質で分類し、移行先を判断する方法
- WinForms ランチャーアプリへの移管を最初のステップとする理由
- 過渡期に RPA をスケジューラとして温存しながらロジックを C# に逃がす戦略
- 最初に移行すべきボットと移行しなくてよいボットの選定基準

## 1. 「RPA が唯一の実行基盤」問題

業務自動化ツールとして RPA が導入されると、スケジュール実行基盤としても使われ始めるケースが多くあります。本来 RPA でなくてよい処理（シンプルなバッチ・ファイル操作・転送処理）までボット化される結果、数百本のボットが増殖し、保守コストが膨らんでいきます。

RPA が普及した背景には「プログラミング不要で自動化できる」という訴求がありましたが、規模が拡大すると以下の問題が顕在化します。

- ボット本数が増えるほど保守・更新のコストが比例して増える
- バージョン管理・テスト・デバッグが通常のソフトウェア開発と比べて難しい
- 処理の複雑化に伴い、RPA の記述力の限界に当たる場面が増える
- 担当者が退職すると引き継ぎが困難になる

## 2. ボットを性質で分類する

移行の前提として、ボットを以下の 3 種類に分類します。分類によって移行先が異なります。

| 種別 | 特徴 | 移行先の候補 |
|---|---|---|
| 手動起動 | 担当者がその場で確認・判断しながら実行 | WinForms デスクトップアプリ |
| スケジュール実行 | 定時・定期に無人で実行 | ジョブスケジューラ + C# exe（別記事参照） |
| Web ブラウザ操作 | 社内 Web システム・SaaS の画面操作 | C# + Playwright（移行対象候補） |
| デスクトップアプリ操作 | Win32・レガシー基幹系の画面操作 | RPA に残す |

Web ブラウザ操作は C# + Playwright でヘッドレス実行・並列処理が可能で、RPA より安定しやすい。ただし UI 変更への設計が必要。詳細は「6_ブラウザ自動化の実装と共通化.md」を参照。

デスクトップアプリ操作（Win32・レガシー基幹系）は RPA の得意領域であるため、無理に移行する必要はありません。

## 3. デスクトップアプリ（ランチャー）への移管

手動起動ボットの移行先として、WinForms でボタンを並べたランチャーアプリが最初のステップとして現実的です。

**基本的な構成:**

- 各ボタンが従来のボット 1 本に対応し、クリックで C# のロジックを実行する
- 処理結果・ログをアプリ内に表示する
- 業務担当者に使ってもらう UI として設計する

**設計のポイント:**

「業務担当者に使ってもらう UI」と「処理ロジック」を分離して設計することが重要です。ロジックを別クラスに切り出しておくと、後からコンソールアプリやサービス化にも転用しやすくなります。

ランチャーの雛形を 1 本作れば、ボタン追加で順次移行できます。最初の 1 本を作る労力が、その後の移行コストを大幅に下げます。

## 4. 過渡期: RPA をスケジューラ兼キックツールとして使う

スケジュール実行ボットをすぐに移行できない場合の現実的な戦略があります。

**やること:**

- RPA 側のボットは「C# の exe を呼び出すだけ」のシンプルな 5〜10 行にする
- 処理ロジックはすべて C# 側に移す

**この戦略のメリット:**

- C# 側はバージョン管理・テスト・デバッグが普通のソフトウェア開発手法で行える
- RPA 側は「スケジューラ」としての機能だけを担う
- ボット本数は変わらないが、中身は空になっていく
- 将来 RPA を解約・縮小する際の判断材料（スケジューラとしての価値だけでコストに見合うか）が明確になる

この戦略を取ることで、RPA ツールのコスト削減をすぐに実現しなくても、開発・保守の問題を先行して解消できます。

## 5. 最初に移行するボットの選定基準

**優先度が高いもの:**

- **数百〜千行規模の複雑なボット**: RPA でこの規模になると保守が困難です。C# に移すと構造が整理でき、テストも書きやすくなります。
- **API 連携・HTTP 通信が主体のもの**: RPA での記述が煩雑になりやすく、C# の方が圧倒的にシンプルに書けます。`HttpClient` を使えば認証・エラーハンドリング・JSON 解析を整理した形で実装できます。
- **手元で確認しながら起動するもの**: デスクトップアプリ化しやすく、担当者の操作感も向上します。

**移行しなくてよいもの（当面）:**

- 5〜10 ステップで完結する単純なボット
- 業務部門ユーザーが自己管理しているボット

移行の優先順位は「保守コストが高いもの」と「C# にすることで明確にメリットが出るもの」を基準にします。単純なボットをむりに移行しても費用対効果が低く、移行リソースを複雑なものに集中させる方が効果的です。
