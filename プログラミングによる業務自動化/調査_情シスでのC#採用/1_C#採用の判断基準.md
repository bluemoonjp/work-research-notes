# C# 採用の判断基準

情シス部門が C# を採用することが適切かどうかの判断軸をまとめます。

## この記事で学ぶこと

- 情シス部門が抱える技術スタックの問題点
- C# という言語の特性と、業務自動化における強み
- 「高機能言語は習得が難しい」という誤解が生まれる理由とその反論
- AI 補助ツールが習得ハードルをどう変えたか
- C# が向いている場面・向かない場面
- 現実的なチーム構成と移行のはじめ方

## 1. 情シス部門の典型的な技術スタック

多くの情シス部門では、以下のようなツールや技術が混在しています。

- **RPA ツール**: Automation Anywhere、UiPath、WinActor など
- **ノーコードツール**: Kintone、Power Automate など
- **VBA**: Excel・Access を中心に使われる業務マクロ
- **レガシー基幹システム**: 独自仕様の帳票や連携スクリプト

これらが混在していると、次のような問題が生じます。

- **ライセンスの散逸**: ツールごとに年間ライセンスが発生し、費用が把握しにくい
- **保守コストの増大**: ツールごとに習熟者が異なり、担当者が離職すると詰まる
- **技術の断片化**: ノウハウが各ツールの独自仕様に蓄積され、横断的に活用できない

## 2. C# という言語の特性

C# は、基幹業務にも耐える堅牢さを持ちながら、しがらみなく設計された現代的な言語です。

- **バランスのよい言語設計**: Java ほど冗長でなく、Python のような実行時型エラーも少ない。型安全でありながらコードが読みやすい。
- **初学者にも読みやすい構文**: 変数・ループ・条件分岐が自然な英語に近い形で書け、VBA や RPA の基礎知識があれば構造を把握しやすい。
- **Windows との親和性**: Microsoft が開発した言語であり、Windows API・COM・Office オートメーションとの連携が容易。
- **単一言語で広範囲をカバー**: デスクトップアプリ・バッチ処理・Web サービス・API 連携・DB アクセスを同じ言語で実装できる。複数ツールを渡り歩く必要がない。
- **技術を一点集中できる**: 複数のノーコードツールをそれぞれ習得するコストと比較すると、1つの汎用言語に投資するほうが長期的に割安になる。
- **LINQ によるデータ操作**: コレクション・データ操作を宣言的・簡潔に記述できる。VBA や RPA で複雑になりがちなデータ加工処理が劇的にシンプルになる。

```csharp
// VBA や RPA のループで書くと数十行になる集計を LINQ で簡潔に書く例
var result = records
    .Where(r => r.Department == "営業部")
    .GroupBy(r => r.Month)
    .Select(g => new { Month = g.Key, Total = g.Sum(r => r.Amount) })
    .OrderBy(x => x.Month)
    .ToList();
```

## 3. 「高機能言語は習得が難しく属人化する」という誤解

「C# は高機能すぎて、一部のエンジニアしか触れない」という懸念がよく聞かれます。しかし実態は逆です。

**高機能言語ほど標準ライブラリや NuGet パッケージで解決できる場面が多い**

CSV 読み込み、HTTP リクエスト、Excel ファイル操作、DB 接続など、業務自動化でよく使う処理の多くはパッケージを1行で呼ぶだけで済みます。「ゼロから実装する」必要がなく、使い方を調べる力があれば十分です。

**低機能ツールでキャパシティを超えると逆に複雑になる**

RPA やノーコードツールで複雑なロジックを組もうとすると、ツールの制約に合わせた回避策が必要になります。結果として「基礎ロジックを自前実装」する羽目になり、工数が増え、品質の個人差が広がります。

| 処理例 | RPA / VBA での対応 | C# での対応 |
|---|---|---|
| データの集計・フィルタリング | ループと条件分岐で数十行 | LINQ で数行 |
| API からデータ取得 | ツール依存のコネクタ設定 | `HttpClient` + JSON デシリアライズ |
| 複数ファイルの一括変換 | ファイルループ + 個別処理 | `Directory.GetFiles` + LINQ |

## 4. AI 補助が状況を変えた

コード生成 AI（GitHub Copilot、Claude、ChatGPT など）の登場により、C# の習得ハードルは数年前と比較にならないほど下がっています。

- **コード生成・サンプル生成**: 「〇〇をするコードを書いて」と質問するだけで動くものが作れる出発点になります。書き方を全部暗記する必要がなくなりました。
- **調査・デバッグ支援**: コンパイルエラーや例外メッセージを貼り付けるだけで、原因と解決策をその場で得られます。
- **コードレビュー**: 書いたコードの問題点・改善案を即座に得られ、独学でも品質が上がります。
- **新しい方法・ライブラリの提案**: 「もっとよい書き方はありますか」と問うだけで、知らなかったライブラリや手法を教えてもらえます。

**「知らないことを知る」コストが激減した**ことで、高機能言語の習得ハードルが大きく下がりました。

また、**ノーコードツールの独自仕様は AI でも調査しにくい**という側面があります。Kintone の特定プラグインの設定方法、特定バージョンの RPA ツールの挙動など、ニッチな仕様は AI の学習データに含まれていないことが多く、結局マニュアルを読み込む必要があります。汎用言語のほうが AI 補助の恩恵が大きいと言えます。

## 5. C# が向いている場面

- **複雑なロジックを持つ RPA ボットの置き換え**: 特に数百〜千行規模のフロー。条件分岐や例外処理が多いものほど C# のほうがシンプルになります。
- **API 連携・ファイル処理・データ変換**: 外部サービスとのデータやり取りや、ファイルの一括変換・加工。
- **バッチ処理**: 夜間・定期実行する業務データの集計・転送・通知処理。
- **基幹システムへのデータ転送・周辺ツール開発**: 基幹システムが公開している API や DB に接続する補助ツール。
- **デスクトップアプリ**: 業務担当者が手元で操作するフォームアプリ（入力 → 処理 → 出力）。

## 6. C# が向かない・置き換えなくてよい場面

すべてを C# に置き換える必要はありません。以下の場合は現状維持が合理的です。

- **業務部門のユーザーが自己改修している単純なボット**: C# に移行すると IT 部門依存になり、ちょっとした修正も IT 部門が対応しなければならなくなります。運用コストが増えるため、むやみに移行しないほうがよいです。
- **Web ブラウザ操作（社内 Web システム・SaaS）**: C# + Playwright も有力な選択肢です。ヘッドレス実行・並列処理・安定性で RPA より優れる場面があります。ただし UI 変更時の保守コストを考慮した設計が必要です。詳細は「5_社内システムへの接続方法の選び方.md」「6_ブラウザ自動化の実装と共通化.md」を参照してください。
- **デスクトップアプリ操作（Win32・レガシー基幹系）**: RPA ツールの方が設定コストが低いことが多く、無理に C# へ移行する必要はありません。
- **完全な一時的な処理**: エンジニアがすぐ捨てる使い捨てスクリプトは、PowerShell や Python で済ませることも選択肢です。

## 7. チーム構成の現実

情シス部門で C# を導入する際の現実的な進め方を示します。

**習得ハードルについて**

RPA 経験者でも VBA の基礎（変数・ループ・条件分岐）があれば、C# の習得ハードルは低めです。概念が共通しているため、「書き方の差」を覚えればよい段階からスタートできます。

**チーム構成の現実解**

- **リード役1人**: 雛形（プロジェクト構成・共通処理・CI/CD など）を整備する担当。C# の経験者か、習得意欲のある人材が1人いれば十分です。
- **改修・追加担当**: 雛形をもとに機能追加や修正を担当する。リード役が整えた土台に沿って書くため、高い習熟度は不要です。

**移行戦略の割り切り**

- 新規開発は C# で行い、既存 RPA は壊れるまで触らない
- 「全部を一気に移行しよう」とすると頓挫しやすい。新規案件から C# を使い始め、実績を積むことが移行を加速させます。
