## ソリューションとプロジェクトの違い

Visual Studio では「ソリューション」と「プロジェクト」の 2 階層でコードを管理する。

```
ソリューション (.sln)
  └── プロジェクト A (.csproj)   ← 実行ファイルや DLL の単位
  └── プロジェクト B (.csproj)
```

| 概念 | ファイル | 役割 |
|------|---------|------|
| ソリューション | `.sln` | 複数プロジェクトをまとめる入れ物。Visual Studio が開く単位 |
| プロジェクト | `.csproj` | ビルドの単位。1 つの実行ファイルまたは DLL を生成する |

初めてコンソールアプリを作成すると、ソリューションとプロジェクトが同じ名前で 1 つずつ自動作成される。

---

## プロジェクトの種類と選び方

新規プロジェクト作成時（**ファイル > 新規作成 > プロジェクト**）に種類を選ぶ。

| テンプレート | 用途 |
|------------|------|
| コンソール アプリ | バッチ処理・自動化ツール。GUIなし |
| Windows フォーム アプリ | ボタン・テキストボックスなどのデスクトップ画面を持つツール |
| クラス ライブラリ | 他プロジェクトから呼び出す共通処理をまとめる DLL |
| WPF アプリ | より高度なデスクトップ UI。XAML で画面を定義する |

業務自動化では **コンソール アプリ** から始めるのが最もシンプル。

---

## 複数プロジェクト構成

規模が大きくなってきたら、役割ごとにプロジェクトを分ける。

**例: Excel 操作ツールの場合**

```
業務ツール.sln
  ├── 業務ツール           (コンソール アプリ) ← エントリポイント
  └── 業務ツール.Common    (クラス ライブラリ) ← 共通ロジック
```

既存ソリューションにプロジェクトを追加するには、**ソリューション エクスプローラー** でソリューションを右クリック → **追加 > 新しいプロジェクト**。

プロジェクト間で参照するには、参照元プロジェクトを右クリック → **追加 > プロジェクト参照**。

## .csproj ファイルの基本

`.csproj` は XML 形式のビルド設定ファイル。SDK スタイル（.NET 5 以降）のものは非常にシンプル。

```xml
<Project Sdk="Microsoft.NET.Sdk">

  <PropertyGroup>
    <OutputType>Exe</OutputType>          <!-- Exe: 実行ファイル / Library: DLL -->
    <TargetFramework>net8.0</TargetFramework>
    <Nullable>enable</Nullable>
  </PropertyGroup>

</Project>
```

NuGet パッケージを追加すると `<ItemGroup>` に `<PackageReference>` が追記される。直接編集も可能だが、通常は Visual Studio の UI から操作する（→ 記事 6）。

---

## よくある落とし穴

- **「プロジェクトがない」エラー**: `.sln` ではなく `.csproj` を直接開いた場合に起こりやすい。常に `.sln` から開くこと
- **フォルダ構成とプロジェクト構成は別物**: ソリューション エクスプローラーの見た目はフォルダ構成と一致しないことがある。プロジェクトの実際の場所はプロパティで確認できる
