# 0: パラダイム概観

## この記事で学ぶこと

- C# がどのような言語パラダイムに属するか

- Java・Python・VBA などの他言語との主な違い
- C# のバージョン進化の概観

## C# のパラダイム

C# は単一のパラダイムに縛られない**マルチパラダイム言語**です。

| パラダイム | 概要 | 主な機能 |
|-----------|------|---------|
| **静的型付け** | 型はコンパイル時に確定する | 型宣言、`var`（型推論）、ジェネリクス |
| **オブジェクト指向** | クラスとオブジェクトを中心に設計する | クラス、継承、インターフェース、ポリモーフィズム |
| **関数型** | 関数を値として扱い、副作用を抑える | ラムダ式、LINQ、`record` 型、イミュータブルな設計 |
| **命令型** | 処理の手順を順番に記述する | for/while ループ、if 文、メソッド呼び出し |

実務では「クラスを中心に設計しつつ、LINQ やラムダ式で関数型スタイルを取り入れる」という書き方が一般的です。

## CLR と .NET との関係

```
C# ソースコード (.cs)
    ↓  コンパイル（csc / Roslyn）
中間言語 (IL: Intermediate Language)
    ↓  JIT コンパイル（CLR: Common Language Runtime）
ネイティブコード（CPU が実行）
```

### マネージドコード

C# のコードは **CLR（Common Language Runtime）** によって管理されます。CLR が提供する主な機能は以下のとおりです。

- **ガベージコレクション（GC）**: 不要なメモリを自動的に解放する。C や C++ のように `free()` や `delete` を手動で呼ぶ必要はありません。
- **型安全性**: メモリへの不正アクセスをランタイムが防ぐ。バッファオーバーランは原則発生しません（`unsafe` ブロックを除く）。
- **例外処理**: 共通の例外機構が言語をまたいで機能する（C# と F# のコードが同じスタックトレースを共有できる等）。

### .NET のバージョン体系

現在の主流は **.NET 6 以降**（旧称 .NET Core）です。旧来の **.NET Framework**（Windows 専用）と区別してください。

| 名称 | 特徴 |
|------|------|
| .NET 6/7/8/9（.NET Core 系） | クロスプラットフォーム（Windows/macOS/Linux） |
| .NET Framework 4.x | Windows 専用。レガシーシステムで使われる |

本シリーズのコードは **.NET 6 以降**を前提とします。

## 他言語との主な違い

### Java との比較

| 観点 | C# | Java |
|------|-----|------|
| プロパティ | 言語組み込み（`get`/`set`）| getter/setter メソッドを手動定義 |
| デリゲート/ラムダ | `delegate`、`Action`/`Func`、ラムダ式 | 関数型インターフェース（`@FunctionalInterface`） |
| 値型 | `struct` で定義可能 | プリミティブ型のみ（`int` 等）、ラッパークラスは参照型 |
| 列挙型 | `enum`（値のみ。メソッドは持てない） | `enum`（メソッドやフィールドを持てる） |
| null 安全 | null 許容参照型（C# 8+、オプトイン） | `@NonNull` アノテーション（外部ツール） |
| LINQ | 言語組み込み | Stream API（Java 8+）で類似操作が可能 |

### Python との比較

| 観点 | C# | Python |
|------|-----|--------|
| 型システム | 静的型付け（コンパイル時に型が確定） | 動的型付け（実行時に型が確定）|
| 実行形式 | コンパイル → JIT → ネイティブ実行 | インタープリタ実行（CPython） |
| インデント | 波括弧 `{}` でブロックを区切る | インデントでブロックを区切る |
| クラス | 厳格なカプセル化（アクセス修飾子） | アクセス制御は慣習（`_` プレフィックス等） |

### VBA との比較

| 観点 | C# | VBA |
|------|-----|-----|
| 実行環境 | CLR / .NET | VBA ランタイム（Office 内） |
| 型安全性 | 強い（`Option Strict` 相当が常に有効） | 弱い（`Variant` 型が広く使われる） |
| GC | あり（CLR が管理） | なし（COM 参照カウント） |
| オブジェクト指向 | クラス・インターフェース・継承をフル活用 | クラスモジュールはあるが機能が限定的 |
| 配列 | 0 始まり固定 | 0 または 1 始まり（`Option Base` で変わる） |

VBA から C# に移行する際の最大の違いは**型安全性の強さ**です。VBA では `Variant` に何でも入れられましたが、C# では型を明示するか、`var` で推論させる必要があります。

## C# のバージョン進化

後続記事で登場するモダン C# の主要機能を概観します。

| バージョン | 主な新機能 |
|-----------|-----------|
| C# 6（2015） | `string` 補間（`$"{}"`）、null 条件演算子（`?.`）、式形式メンバー |
| C# 7（2017） | タプル、パターンマッチング（`is`）、`out` 変数 |
| C# 8（2019） | switch 式、null 許容参照型、非同期ストリーム、using 宣言 |
| C# 9（2021） | レコード型（`record`）、トップレベルステートメント、`init` セッター |
| C# 10（2022） | グローバル using、ファイルスコープ名前空間、レコード構造体 |
| C# 12（2024） | プライマリコンストラクタ、コレクション式 |

本シリーズでは **C# 8 以降**のモダンな書き方を積極的に紹介します。

## よくある落とし穴

- **.NET Framework と .NET（Core）の混同**: 検索結果が .NET Framework 向けの古い情報である場合があります。`System.IO.File` のような基本クラスは共通ですが、`HttpClient` の使い方など細部が異なります。
- **静的型付けの厳格さ**: Python のように「とりあえず実行して型エラーを確認する」スタイルは C# では機能しません。コンパイルエラーが先に出るため、型を意識した設計が必要です。
