# SAML 強制環境での API アクセス回避策

## 状況の整理

cybozu.com で「SAML 認証の使用を必須にする」設定を有効にすると、
一般ユーザーはパスワード認証で REST API を実行できなくなる。

影響を受ける認証方式:

| 認証方式 | 一般ユーザー | cybozu.com 共通管理者 |
|---------|------------|----------------------|
| パスワード認証（kintone / ガルーン） | **不可** | 可 |
| API トークン認証（kintone のみ） | **制限なし** | **制限なし** |
| OAuth 認証（kintone / ガルーン） | **制限なし** | **制限なし** |
| セッション認証 | **制限なし** | **制限なし** |

出典:
- [認証 - cybozu developer network（kintone）](https://cybozu.dev/ja/kintone/docs/rest-api/overview/authentication/)
- [認証 - cybozu developer network（ガルーン）](https://cybozu.dev/ja/garoon/docs/rest-api/overview/authentication/)

---

## 回避策 1: OAuth 認証を使う（推奨）

SAML 強制環境でも OAuth は制限されない。

**メリット:**
- 一般ユーザー権限で API を実行できる
- リフレッシュトークンが無期限のため、長期バッチ処理に向く
- kintone・ガルーン両方で利用可能

**デメリット:**
- 初回の認可コード取得にブラウザ操作（ユーザーの認可）が必要
- サービスアカウント的な完全無人実行には向かない

**対処:** 初回だけ手動でリフレッシュトークンを取得し、以降はリフレッシュトークンでアクセストークンを自動更新する運用にする。

→ 詳細: `kintone_OAuth2.0認証.md`

---

## 回避策 2: kintone API トークンを使う（kintone のみ）

kintone には「API トークン」機能がある。SAML 強制の影響を受けない。

**メリット:**
- 完全に無人での実行が可能
- アプリ単位で発行・管理できる

**デメリット:**
- **kintone 専用**。ガルーンには使えない
- アプリごとに発行が必要
- ユーザー単位のアクセス制御ではなくアプリ単位

**設定場所:** kintone アプリの設定 →「API トークン」

出典: [APIトークンを使ってみよう - cybozu developer network](https://cybozu.dev/ja/kintone/tips/development/customize/development-know-how/api-tokens/)

---

## 回避策 3: cybozu.com 共通管理者アカウントを API 専用に使う

SAML 強制環境でも、cybozu.com 共通管理者はパスワード認証で API を実行できる。

**手順:**
1. API アクセス専用の cybozu.com 共通管理者アカウントを作成する
2. そのアカウントでパスワード認証を使って API を呼び出す

**注意点:**
- 共通管理者権限は強力すぎる。必要最低限の権限設定に注意が必要
- cybozu.com の仕様変更・制限強化のリスクがある
- **推奨度は低い**。権限最小化の観点から OAuth の利用を優先すること

---

## 回避策 4: SAML 除外 URL でのパスワード認証

SAML 認証を無効にしてログインするための特殊 URL が存在する。

```
https://{サブドメイン}.cybozu.com/login?saml=off
```

この URL を使うと、SAML を迂回してサイボウズ標準のパスワード認証でログインできる。

**重要な制限:**
- 「SAML 認証の使用を必須にする」設定を有効にしている場合、
  この URL でパスワード認証できるのは **cybozu.com 共通管理者のみ**。
- 一般ユーザーは SAML 強制環境ではこの URL を使っても
  パスワード認証でログインできない。

出典: [SAML認証を無効にして管理画面にアクセスする方法 - cybozu developer community](https://developer.cybozu.io/hc/ja/community/posts/200879764-SAML%E8%AA%8D%E8%A8%BC%E3%82%92%E7%84%A1%E5%8A%B9%E3%81%AB%E3%81%97%E3%81%A6%E7%AE%A1%E7%90%86%E7%94%BB%E9%9D%A2%E3%81%AB%E3%82%A2%E3%82%AF%E3%82%BB%E3%82%B9%E3%81%99%E3%82%8B%E6%96%B9%E6%B3%95)

## SAML 除外できるユーザーの設定方法

cybozu.com では **ユーザー単位で SAML 認証を除外する設定はできない**（2025 年時点）。

SAML 除外が可能なのは以下のみ:

| 対象 | 内容 |
|------|------|
| cybozu.com 共通管理者 | SAML 強制でもパスワード認証が可能（仕様上の例外） |
| SAML 設定そのものを無効化 | 全ユーザーに影響するため、運用上は困難 |

一般ユーザーを特定の IP アドレスからのアクセスのみ SAML 除外する機能などは確認されていない。

---

## IP アドレス制限との組み合わせ

cybozu.com はアクセス元 IP アドレスによる制限機能（セキュリティ強化設定）を持つ。

**想定される組み合わせ:**
- 社内ネットワーク（特定 IP）からのアクセスのみ許可
- その環境内のバッチサーバーから OAuth または API トークンで API を呼び出す

**注意:** IP 制限はログイン時の制御であり、認証方式そのものの SAML 除外とは異なる概念。
SAML 強制設定と IP 制限は独立した設定項目。

---

## 推奨構成まとめ

```
SAML 強制環境でのプログラム連携

kintone のみ:
  └─ API トークン認証（最もシンプル・無人実行可能）

kintone + ガルーン 両方 または ユーザー権限が必要:
  └─ OAuth 2.0 認証
       ├─ 初回: ブラウザで認可 → リフレッシュトークン取得
       └─ 以降: リフレッシュトークンでアクセストークンを自動更新
```
